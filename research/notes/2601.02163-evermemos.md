# EverMemOS: A Self-Organizing Memory Operating System for Structured Long-Horizon Reasoning

- **arXiv id:** 2601.02163
- **Date:** 2026-01
- **URL:** https://www.arxiv.org/abs/2601.02163
- **Taxonomy tags:** Factual / Token-level / Formation + Evolution + Retrieval
- **Authors:** Hu, Gao, Zhou, Xu, Bai, et al. (EverMind, Shanda Group)

## Problem

Long-term interactive agents need memory that consolidates evolving user
state and resolves conflicts, not just stores isolated records and
retrieves fragments — most existing systems do the latter and struggle
with the former.

## Mechanism

EverMemOS implements an "engram-inspired lifecycle" in three stages:

1. **Episodic Trace Formation** — dialogue streams become `MemCells`
   capturing episodic traces, atomic facts, and **time-bounded "Foresight"
   signals** (facts with a known future relevance window, not just facts
   with a past timestamp — e.g. "the meeting is next Friday").
2. **Semantic Consolidation** — `MemCells` are organized into thematic
   `MemScenes`, distilling stable structures and updating user profiles.
3. **Reconstructive Recollection** — retrieval is `MemScene`-guided
   agentic composition of "necessary and sufficient" context, not a flat
   top-k similarity dump.

State-of-the-art on LoCoMo and LongMemEval; also reports a profiling study
on PersonaMem v2.

## What's reusable for corticore

Two of the three stages map onto features corticore already has in a
lighter-weight, non-graph form:

- `MemCells -> MemScenes` (Semantic Consolidation) is the same operation as
  corticore's `reflect()` merge/dedup pass in
  [`dynamics/consolidate.py`](../../src/corticore/dynamics/consolidate.py)
  — a `MemScene` is roughly "a cluster of memories that got merged," which
  corticore represents via `superseded_by` links rather than a first-class
  scene node. No change needed; this validates the existing design.
- **"Foresight" time-bounded facts are genuinely new and cheap to adopt.**
  corticore's only forgetting mechanism today is *access-based decay*
  (things fade if unused). Foresight adds a second, orthogonal forgetting
  trigger: a fact can carry a known expiry regardless of how often it's
  accessed (e.g. "standup moved to 10am this week" should stop being
  recalled after the week ends, even if it was checked constantly during
  it). This requires no new infrastructure — one nullable field plus a
  check in the existing `reflect()` pass.

## Setup cost

Zero. `expires_at` is an optional field with no default behavior change for
callers who don't set it; fully compatible with the zero-setup default
path and every existing store/embedder.
