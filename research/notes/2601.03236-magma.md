# MAGMA: A Multi-Graph based Agentic Memory Architecture for AI Agents

- **arXiv id:** 2601.03236
- **Date:** 2026-01
- **URL:** https://arxiv.org/abs/2601.03236
- **Taxonomy tags:** Factual / Token-level / Evolution + Retrieval
- **Authors:** Jiang, Li, Li, Li (UT Dallas, University of Florida)

## Problem

Memory-Augmented Generation systems typically store memory as a monolithic,
similarity-searchable pool, entangling temporal, causal, and entity
information together. That makes retrieval opaque (why was this retrieved?)
and misaligned with query intent (a query about *when* something happened
retrieves the same way as a query about *who* was involved).

## Mechanism

MAGMA represents each memory item across **four orthogonal graphs**:
semantic, temporal, causal, and entity. Retrieval is reframed as
policy-guided traversal over these relational views — an "Adaptive
Traversal Policy" routes a query to the relevant graph(s), prunes
irrelevant regions, and fuses the resulting subgraphs into context. Notably
for corticore, MAGMA also uses a **dual-stream memory evolution**
mechanism: latency-sensitive event ingestion is decoupled from
asynchronous structural consolidation, so writing a memory stays fast while
consolidation happens separately. Outperforms prior systems on LoCoMo and
LongMemEval with lower latency and token use.

## What's reusable for corticore

The **dual-stream pattern is already corticore's architecture**:
`Memory.remember()` is the fast, synchronous ingestion path (write once,
return immediately — no ranking or consolidation work happens inline), and
`Memory.reflect()` is the separate, explicitly-invoked structural
consolidation pass (dedup, merge, conflict resolution, pruning — see
[`dynamics/consolidate.py`](../../src/corticore/dynamics/consolidate.py)).
This paper is independent validation that separating these concerns is the
right shape, not a paper that requires new code.

The four-orthogonal-graph representation itself is not adopted: it assumes
a graph-capable store and a traversal policy, neither of which fits this
round's SQL-only backend work (`PostgresStore`). If corticore later adds a
graph backend (e.g. for CompassMem-style relational retrieval), MAGMA's
"orthogonal relation types" framing would be worth revisiting together with
it.

## Setup cost

High for the full architecture (graph storage + traversal policy). The
already-adopted dual-stream pattern had zero incremental cost since it was
independently arrived at in v0.1.
