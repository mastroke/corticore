# MemRL: Self-Evolving Agents via Runtime Reinforcement Learning on Episodic Memory

- **arXiv id:** 2601.03192
- **Date:** 2026-01
- **URL:** https://arxiv.org/abs/2601.03192
- **Taxonomy tags:** Experiential / Working / Evolution + Retrieval
- **Authors:** Zhang, Wang, Zhou, Liao, Feng, et al. (Shanghai Jiao Tong University, Xidian University, NUS, MemTensor)

## Problem

LLM agents can't easily self-evolve the way humans do via "Constructive
Episodic Simulation" (retrieving past experience to synthesize solutions to
novel tasks): fine-tuning is expensive and forgets catastrophically, and
existing memory-based methods rely on passive semantic matching that often
retrieves noise — a similar-looking past episode isn't necessarily a
*useful* one.

## Mechanism

MemRL keeps the LLM frozen and treats episodic memory as the only "plastic"
(learnable) part of the system. It uses a **Two-Phase Retrieval** mechanism:
first filter candidate memories by semantic relevance (cheap, as usual),
then re-rank the survivors by a **learned Q-value (utility)** that is
continuously refined via environmental feedback (trial-and-error, i.e.
runtime reinforcement learning) — not by embedding similarity alone. This
lets the agent tell "high-value strategy" apart from "similar-looking
noise" without any weight updates to the LLM itself. Evaluated on HLE,
BigCodeBench, ALFWorld, and Lifelong Agent Bench against memory-augmented
baselines.

## What's reusable for corticore

The two-phase shape — *filter by relevance, then re-rank by a utility
signal distinct from relevance* — is exactly what
[`dynamics/retrieval.py`](../../src/corticore/dynamics/retrieval.py) already
does, just with a much cheaper utility signal: our decay-adjusted score
multiplies keyword+embedding relevance by a **hand-designed** salience/decay
factor instead of a **learned** Q-value. MemRL is evidence that "relevance
is not the same as usefulness" is worth taking seriously — it validates the
retrieval module's shape without requiring us to adopt RL.

## Setup cost

High if adopted literally: online RL requires an environment/reward signal
loop, training infrastructure, and persistent Q-value storage per
memory-task pair. None of this fits the zero-setup default, and there's no
"optional extra" that reduces this — it's an architectural commitment, not
a pip install.
